# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

**LGTMHub** - 安心安全なLGTM画像共有サービス

### プロダクトコンセプト

- **視認性の高いLGTM文字**: 白文字+黒縁固定で、背景画像を問わずLGTMの文字が必ず読める
- **ログイン必須の投稿**: GitHub OAuth認証により乱用・不適切コンテンツを抑制し、会社でも安心して開ける
- **重複排除**: 画像ハッシュで同一画像の重複登録を防ぎ、一覧の多様性を維持する

### プロダクトビジョン

コードレビューのApprove時に、気の利いたLGTM画像を手軽に貼れる場を提供する。
lgtmoon.comの課題（白文字の見づらさ・不適切コンテンツ・同一画像の氾濫）を解決し、チームの雰囲気を和ませる安心安全なサービスを実現する。
マークダウンリンクのワンクリックコピーで、レビュアーのワークフローを一切妨げない。

### 目的

- コードレビュー文化をより楽しいものにする
- 不適切コンテンツを排除し、業務環境でも安心して利用できるサービスを提供する
- 同一画像の大量登録を防ぎ、多様なLGTM画像を維持する

## ターゲットユーザー

### プライマリーペルソナ: 田中一郎（29歳、フリーランスエンジニア）

- 開発歴5年以上、性格は穏やか
- GitHub・VS Code・ターミナル中心の開発環境
- **課題**: コードレビュー時のApproveがいつも「LGTM」のテキストのみ、または素っ気なくコメントなし
- **欲しいもの**: クールな画像でLGTMしたい。よく使う画像はお気に入り登録したい
- **ワークフロー**: PRレビュー時にサイトを開き、好みの画像のマークダウンリンクをコピーしてApproveコメントに貼る

### セカンダリーペルソナ: 佐藤花子（35歳、スタートアップのテックリード）

- 開発歴10年以上、2〜5名のチームをマネジメント
- **課題**: lgtmoon.comが好きだが、過去にアダルト画像を貼ってしまった経緯があり会社から再発防止を求められている
- **欲しいもの**: 会社で開いても安心なサイト。かわいいLGTM画像でメンバーをほっこりさせたい。自分でも画像を登録したい
- **ワークフロー**: チームメンバーにサービスを共有し、安心して使える場として定着させる

## 成功指標（KPI）

### プライマリーKPI

| 指標 | 計測開始 | 初期目標（仮） | 計測ツール |
|------|----------|----------------|------------|
| 月間アクティブユーザー数（MAU） | ローンチ翌月から | ローンチ3ヶ月後に **20 MAU** | Vercel Analytics（ユニーク訪問者数で代用） |
| 1ユーザーあたりのMarkdownコピー回数 | ローンチ翌月から | 全MAUの **50%** が週1回以上コピー | Vercel Analytics カスタムイベント（週次のユニーク発火ユーザー数を手動集計） |

**KPI計測の補足**:
- **MAU**: 厳密なユーザーIDベースの計測ではなく、Vercel Analytics のユニーク訪問者数（IP+UA ベース）で代用する。ログイン/未ログインを問わず合算するため、実数とは誤差が生じる前提とする
- **Markdownコピー回数**: 各コピー操作で `markdown_copied` カスタムイベントを発火し、Vercel Analytics ダッシュボードからイベント発火ユーザー数を週次で手動集計する。「全MAUの50%」の分母は同期間の MAU、分子は1度以上イベントを発火したユニーク訪問者数とする

### セカンダリーKPI

| 指標 | 計測開始 | 初期目標（仮） | 計測ツール |
|------|----------|----------------|------------|
| 登録画像数の推移 | ローンチ初月から | 月次10%以上の増加 | Supabase クエリ（管理ダッシュボード） |
| お気に入り登録数 | ローンチ翌月から | お気に入りを1件以上登録したユーザーあたり平均5件以上 | Supabase クエリ |
| 不適切コンテンツの通報・削除件数 | P1機能リリース後から | 月次の通報件数<削除件数の50%（誤通報率の管理） | 管理者ダッシュボード |
| トップページの直帰率 | ローンチ初月から | 70%以下（業界平均的な目安値、MVPローンチ後に妥当性を検証） | Vercel Analytics（`/` ページの直帰率） |

**セカンダリーKPI計測の補足**:
- **お気に入り登録数**: 分母は「期間中に1件以上のお気に入りを登録したユーザー数」とする（使っていないユーザーを分母から除外）
- **直帰率**: 計測対象はトップページ（`/`）のみ。Vercel Analytics の bounce rate を参照する

> 仮目標値はローンチ前のたたき台。ローンチ後1〜3ヶ月の実データを見て確定値へ更新する。
> KPIの確定後はこのテーブルを更新し、変更履歴を残す。

## 機能要件

### P0: MVPで必須の機能

#### 1. 画像登録機能

**ユーザーストーリー**:
ログイン済みユーザーとして、LGTM画像のバリエーションを増やすために、Web上の画像URLを登録してLGTM文字を合成した画像を作成したい

**受け入れ条件**:
- [ ] 画像URLを入力フォームに入力して登録できる
- [ ] サーバーサイドで画像をダウンロードし、白文字+黒縁の「LGTM」文字を合成する（Sharpを使用）
- [ ] 合成後の画像をWebP形式に変換し、長辺 400px にリサイズ（元アスペクト比保持・原画 < 400px は拡大しない）してVercel Blobに保存する
- [ ] 対応フォーマット: JPEG・PNG・GIF（静止画のみ）、元ファイル最大10MB
- [ ] pHashによる重複チェックを行い、既存画像と一致する場合は作成せず既存画像ページへリダイレクトする
- [ ] 登録制限: 1ユーザーあたり1日10枚まで（超過時はエラーメッセージを表示）
- [ ] 登録処理は非同期で行い、処理中インジケーターを表示し、完了後に通知する
- [ ] 登録処理は10秒以内に完了する（Vercel Functionのタイムアウト内）

**優先度**: P0（必須）

---

#### 2. 画像削除機能

**ユーザーストーリー**:
ログイン済みユーザーとして、誤って登録した画像や不要になった画像を管理するために、自分が登録したLGTM画像を削除したい

**受け入れ条件**:
- [ ] 自分が登録した画像の詳細ページに「削除」ボタンが表示される
- [ ] 削除前に確認ダイアログを表示する
- [ ] 削除後は画像のステータスが `deleted` に更新され、画像一覧・お気に入り一覧から非表示になる（論理削除）
- [ ] 他のユーザーの画像には削除ボタンが表示されない

**注**: Vercel Blob からの物理削除は誤操作・復旧不能リスクを考慮し、P1フェーズ（後述「8. 削除画像の物理クリーンアップ」）で実装する。MVPでは Blob URL 直アクセスは可能な状態を許容する。

**優先度**: P0（必須）

---

#### 3. GitHub OAuth認証

**ユーザーストーリー**:
エンジニアとして、新しいアカウントを作らずに使い始めるために、GitHubアカウントでログインしたい

**受け入れ条件**:
- [ ] GitHubアカウントでOAuth認証ができる
- [ ] ログイン後、GitHubのアバター画像と表示名をプロフィールとして使用する
- [ ] ログインなしでも画像の一覧閲覧・マークダウンリンクのコピーができる
- [ ] 画像登録はログイン必須で、未ログイン時は「ログインして登録」ボタンを表示する
- [ ] セッションはSupabaseで管理する

**優先度**: P0（必須）

---

#### 4. お気に入り機能（登録・解除・一覧）

> **概要**: 「お気に入り登録・解除」と「お気に入り画像一覧」は同一ドメインに属する機能だが、画面とAPIエンドポイントが独立しているため、サブ機能 4-A / 4-B として明示する。両方を満たして初めて機能4の完了とみなす。

##### 4-A. お気に入り登録・解除

**ユーザーストーリー**:
ログイン済みユーザーとして、よく使う画像にすぐアクセスするために、気に入った画像をお気に入りに追加したい

**受け入れ条件**:
- [ ] 任意の画像（他ユーザー登録のものも含む）をお気に入りに追加・解除できる
- [ ] お気に入りリストは自分のみ閲覧可能（非公開）
- [ ] お気に入り状態はページをまたいでも保持される

##### 4-B. お気に入り画像一覧画面

**ユーザーストーリー**:
ログイン済みユーザーとして、好みの画像をすぐに見つけるために、お気に入り登録した画像だけを一覧表示したい

**受け入れ条件**:
- [ ] お気に入りした画像のみをフィルタリングして一覧表示する
- [ ] 一覧からマークダウンリンクをコピーできる
- [ ] お気に入りが0件の場合は適切なempty stateを表示する

**優先度**: P0（必須）

---

#### 5. 画像一覧画面

**ユーザーストーリー**:
すべてのユーザー（未ログインを含む）として、使いたいLGTM画像を見つけるために、登録済みのLGTM画像を一覧で閲覧したい

**受け入れ条件**:
- [ ] 登録済みのLGTM画像を新着順（登録日時の降順）で一覧表示する
- [ ] ログインなしでも閲覧できる
- [ ] 各画像にマークダウンコピーボタンを表示し、クリック一発でマークダウンリンクをクリップボードにコピーできる
- [ ] コピー後にフィードバック（「コピーしました」等）を表示する
- [ ] 1ページあたり最大20件のカーソルページネーションで取得する（詳細は `docs/functional-design.md#画像一覧取得` 参照）
- [ ] 50件以上のデータがあっても一覧描画が破綻しないこと（パフォーマンス基準は下記参照）
- [ ] パフォーマンス基準は「非機能要件 > パフォーマンス」セクションを参照（LCP 3秒以内、Vercel Edge Network経由・3G相当回線で測定）

**優先度**: P0（必須）

### P1: 重要だが後回し可能な機能

#### 6. 管理者による画像削除機能

**ユーザーストーリー**:
管理者として、不適切なコンテンツを速やかに排除しサービスの品質を維持するために、任意のユーザーの画像を即時削除したい

**受け入れ条件**:
- [ ] 管理者ロール（`user_profiles.is_admin = true`）を持つユーザーのみが任意の画像を削除できる
- [ ] 削除後は画像のステータスが `deleted` に更新され、画像一覧・お気に入り一覧から非表示になる（論理削除）
- [ ] 管理者削除の場合は **Vercel Blob からも即時物理削除する**（機能8の30日待機ルールの例外。不適切コンテンツの拡散を防ぐため）
- [ ] 管理者操作のログを記録する（操作者ID・対象画像ID・実行日時）

**優先度**: P1（重要）

---

#### 7. ユーザー通報機能

**ユーザーストーリー**:
ログイン済みユーザーとして、不適切なコンテンツをなくすために、問題のある画像を通報したい

**受け入れ条件**:
- [ ] 各画像に「通報」ボタンを表示する
- [ ] 通報カテゴリを選択できる（不適切コンテンツ・著作権侵害など）
- [ ] 通報後、管理者に通知が届く
- [ ] 同一画像が **5件以上の通報** を受けた場合、自動的に非表示（レビュー待ち）にする
- [ ] 同一ユーザーが同一画像に対して通報できるのは1回のみ
- [ ] 閾値（5件）はMVP後に実データを見て調整可能とする

**優先度**: P1（重要）

---

#### 8. 削除画像の物理クリーンアップ

**ユーザーストーリー**:
運用者として、ストレージコストと未公開コンテンツの拡散リスクを抑えるために、論理削除済み画像のBlob実体を一定期間後に物理削除したい

**受け入れ条件**:
- [ ] `status = 'deleted'` かつ `deleted_at` から30日経過した画像のBlobを物理削除する
- [ ] 物理削除はGitHub Actionsの日次クリーンアップジョブで実行する
- [ ] 物理削除完了後、`lgtm_images` テーブルから該当レコードを削除する
- [ ] 削除失敗時はリトライ可能とし、3回失敗で運用者へ通知する

**優先度**: P1（重要）

**備考**: MVPの機能2（ユーザー画像削除）は論理削除のみで完結する。本機能は論理削除と分離して運用負荷を最小化する目的で別建てとする。なお機能6（管理者削除）は不適切コンテンツの即時排除を優先するため本機能の30日待機ルールの例外として即時物理削除を行う。

---

#### 9. ファイルアップロード対応

**ユーザーストーリー**:
ログイン済みユーザーとして、手元の画像を使いたいために、ローカルファイルをアップロードして画像を登録したい

**受け入れ条件**:
- [ ] ドラッグ&ドロップまたはファイル選択でローカル画像をアップロードできる
- [ ] ファイルサイズ上限: 10MB
- [ ] 対応フォーマット: JPEG・PNG
- [ ] pHashによる重複チェックをURL登録（機能1）と同じロジックで実施する
- [ ] 日次登録上限（10枚/日）はURL登録・ファイルアップロード合算でカウントする
- [ ] 登録処理は機能1と同じ非同期フロー（処理中インジケーター→完了通知）で行う
- [ ] エラー時はURL登録と同等のメッセージ（フォーマット不可・サイズ超過・重複・上限到達）を表示する

**優先度**: P1（重要）

## 非機能要件

### パフォーマンス

| 操作 | 目標 | 測定指標・条件 |
|------|------|----------------|
| 画像一覧の初期表示（20件） | 3秒以内 | LCP（Largest Contentful Paint）、Vercel Edge Network経由・3G相当回線、Lighthouse / Vercel Analyticsで計測 |
| 画像詳細ページの初期表示 | 2秒以内 | LCP、Vercel Edge Network経由・3G相当回線、Lighthouseで計測 |
| マークダウンリンクのコピー | 100ms以内 | クリックからクリップボード書き込み完了までの計測 |
| 画像登録処理（ダウンロード・合成・CDN保存） | 10秒以内 | Vercel Functionのタイムアウト内、サーバーログでstart/end計測 |
| API応答（一覧取得） | 500ms以内（p95） | TTFB相当、Vercel Functionで計測、Vercel Analytics で取得 |

> パフォーマンス指標は `docs/architecture.md`「パフォーマンス要件」と同期させる。変更時は両ドキュメントを同時に更新する。

### ユーザビリティ

- 初めて使うユーザーが5分以内に基本操作（閲覧・コピー）を習得できる
  - **検証方法**: Playwright E2Eテストで「未ログインユーザーがトップページ到達 → 任意の画像のマークダウンをコピー」までを **5ステップ以内**（ページ遷移なしで完結）で実現することで担保する
- ログインなしでも閲覧・コピーが可能（導入障壁を低く保つ）
- エラーメッセージは原因と対処法を日本語で明示する
- PCファースト設計、レスポンシブ対応はするがモバイル最適化は後回し

### 信頼性

- 登録画像データは自動バックアップにより消失しない（Supabaseの自動バックアップを利用）
- 画像登録が途中で失敗した場合、アップロード済みのファイルはロールバックする
- 危険な操作（削除）には確認ダイアログを必ず表示する

### セキュリティ

- 画像登録はGitHub OAuthログイン済みユーザーのみ可能
- 他ユーザーの画像の削除はAPIレベルで拒否する
- Supabase Row Level Security（RLS）でデータアクセスを制御する
- 外部URLからの画像取得はSSRF対策としてプライベートIPへのアクセスを禁止する

### スケーラビリティ

- Vercel・Supabaseの無料枠を最大限活用し、初期コストゼロで運用する
- 収益化はMVP成立後に検討する

- OGP・SNSシェア対応（MVP外）

- 多言語対応（日本語のみ）
