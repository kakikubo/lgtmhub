# 要求: 登録画像のサイズを長辺 800px にリサイズし直す

GitHub issue: https://github.com/kakikubo/lgtmhub/issues/75

## 背景

issue #52 で合成画像を 266×199 固定 (中央クロップ) に変更したが、運用してみたところ詳細ページの見栄えが貧弱になった。LGTM 画像は PR コメントへの貼付だけでなくサイト上での閲覧体験も同等に重要であり、本 issue では合成画像を **長辺 800px・元アスペクト比保持** に変更する。

## 達成したいこと

- **A. 詳細ページの見栄え向上** (面積 266×199 → 長辺 800、約 9 倍)
- **B. PR コメント側の互換性維持** (GitHub PR コメント実効幅 約 720px と同等)
- **C. 縦長画像にも自然対応** (中央クロップを撤廃し原画アスペクト比を保持)

## 仕様

| 項目 | 仕様 |
|---|---|
| リサイズ方式 | 長辺 cap・元アスペクト比保持 (`fit: 'inside'` 相当) |
| 上限値 | 長辺 800px |
| 拡大方針 | 原画 < 800 は拡大しない (原画サイズのまま保存) |
| 出力フォーマット | WebP (現状維持、`WEBP_QUALITY = 85` 維持) |
| LGTM 文字サイズ | 既存の `Math.max(24, Math.floor(canvasWidth * 0.15))` を維持 |
| トップページグリッド | `aspect-[4/3] object-cover` 維持 (CSS で中央クロップ) |
| 詳細ページの表示 | 親 `max-w-3xl` 維持、内側を `max-w-[800px]` に拡張 (実効最大幅 736px) |
| pHash の互換性 | 元画像 buffer に対して計算しており壊れない |

## リサイズ後サイズの例

| 入力 | 出力 |
|---|---|
| 1920×1080 (16:9) | 800×450 |
| 1200×900 (4:3) | 800×600 |
| 1000×1000 (正方形) | 800×800 |
| 736×1000 (縦長) | 588×800 |
| 600×400 (原画 < 800) | 600×400 (拡大しない) |

## 受け入れ条件

- 新規登録画像が長辺 800px・元アスペクト比保持で保存される (DB の `width`/`height` も一致)
- 1920×1080 → 800×450 / 1000×1000 → 800×800 / 736×1000 → 588×800 になる
- 原画が 800 未満 (例 600×400) の場合、そのままのサイズで保存される (拡大しない)
- 既存 266×199 画像はそのまま閲覧・コピーできる (Blob URL を破壊しない)
- 詳細ページで画像が実効 736px 幅で表示される (はみ出さない)
- トップページグリッドで新旧画像が混在しても 4:3 カードに整然と並ぶ
- 重複検知 (pHash) が機能し、新仕様でも既存画像と同一画像なら 409 を返す
- PRD・機能設計書・開発ガイドライン・用語集の数値表記が長辺 800px で整合する
- `npm run lint` / `npm run typecheck` / `npm test` がすべて通る
- 既存 E2E (`image-register.test.ts` / `image-list.test.ts` / `image-detail.test.ts`) が通る

## スコープ外

- 既存 266×199 画像の再合成 (Blob URL 破壊リスクのため別 issue)
- トップページグリッドのアスペクト比変更
- 詳細ページ親幅の拡張 (`max-w-3xl` → `max-w-4xl`)
- LGTM 文字サイズの全画像統一化
